# Dependency Management with Maven

---

## What is Maven?

Maven is a **build automation and dependency management tool** for Java projects. It automates the process of compiling code, managing libraries, running tests, and packaging applications.

Instead of manually downloading JAR files and adding them to your project, Maven automatically downloads and manages everything for you.

### What Maven does:
- Downloads libraries your project needs (dependencies)
- Compiles your source code
- Runs tests
- Packages your project into a JAR or WAR file
- Manages project structure and build lifecycle

---

## Why Use Maven?

| Problem without Maven | Maven Solution |
|---|---|
| Manually download JAR files | Maven downloads them automatically from repositories |
| JAR version conflicts | Maven manages versions and resolves conflicts |
| Different project structures | Maven enforces a standard project layout |
| Complex build scripts | Maven uses a single `pom.xml` file |
| Missing transitive dependencies | Maven automatically downloads all required dependencies |

## Project Structure (Standard Maven Layout)

Maven expects a specific folder structure:

```
my-project/
├── pom.xml                          ← Project configuration file
├── src/
│   ├── main/
│   │   ├── java/                    ← Your Java source code
│   │   │   └── com/myapp/
│   │   │       └── Main.java
│   │   └── resources/               ← Configuration files, properties
│   │       └── application.properties
│   └── test/
│       ├── java/                    ← Test code
│       │   └── com/myapp/
│       │       └── MainTest.java
│       └── resources/               ← Test configuration
└── target/                          ← Compiled output (generated by Maven)
    ├── classes/
    └── my-project-1.0.jar
```

This standard structure means any Java developer can open any Maven project and immediately know where everything is.

## pom.xml (Project Object Model)

The `pom.xml` is the **heart of every Maven project**. It is an XML file that contains all information about the project and its configuration.

### Complete pom.xml example:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
         http://maven.apache.org/xsd/maven-4.0.0.xsd">

    <!-- Model version -->
    <modelVersion>4.0.0</modelVersion>

    <!-- Project coordinates (GAV) -->
    <groupId>com.myapp</groupId>
    <artifactId>my-project</artifactId>
    <version>1.0.0</version>
    <packaging>jar</packaging>

    <!-- Project information -->
    <name>My Java Project</name>
    <description>A simple Java application</description>

    <!-- Properties (versions managed in one place) -->
    <properties>
        <maven.compiler.source>17</maven.compiler.source>
        <maven.compiler.target>17</maven.compiler.target>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
        <junit.version>5.10.0</junit.version>
    </properties>

    <!-- Dependencies (libraries your project needs) -->
    <dependencies>
        <!-- JUnit for testing -->
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <version>${junit.version}</version>
            <scope>test</scope>
        </dependency>

        <!-- MySQL Connector -->
        <dependency>
            <groupId>mysql</groupId>
            <artifactId>mysql-connector-java</artifactId>
            <version>8.0.33</version>
        </dependency>

        <!-- Gson for JSON -->
        <dependency>
            <groupId>com.google.code.gson</groupId>
            <artifactId>gson</artifactId>
            <version>2.10.1</version>
        </dependency>
    </dependencies>

    <!-- Build configuration -->
    <build>
        <plugins>
            <!-- Maven Compiler Plugin -->
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-compiler-plugin</artifactId>
                <version>3.11.0</version>
                <configuration>
                    <source>17</source>
                    <target>17</target>
                </configuration>
            </plugin>

            <!-- Maven JAR Plugin (for executable JAR) -->
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-jar-plugin</artifactId>
                <version>3.3.0</version>
                <configuration>
                    <archive>
                        <manifest>
                            <mainClass>com.myapp.Main</mainClass>
                        </manifest>
                    </archive>
                </configuration>
            </plugin>
        </plugins>
    </build>

</project>
```

## Maven Coordinates (GAV)

Every project and dependency in Maven is identified by three values called **GAV coordinates**:

| Coordinate | Meaning | Example |
|---|---|---|
| **groupId** | Organization or company name | `com.google.code.gson` |
| **artifactId** | The name of the project/library | `gson` |
| **version** | The version number | `2.10.1` |

Together they uniquely identify any library in the Maven world:

```
com.google.code.gson:gson:2.10.1
   groupId          artifactId  version
```

## Maven Repository

A Maven repository is a **central storage location** where JAR files and their metadata are stored. Maven downloads dependencies from these repositories automatically.

### Types of Repositories:

| Repository | Description | URL |
|---|---|---|
| Local Repository | Cache on your machine (`~/.m2/repository`) | N/A |
| Central Repository | Default public repository maintained by Maven | https://repo1.maven.org/maven2 |
| Remote Repository | Private company or custom repositories | Company-specific URLs |

### How it works:
```
1. You declare a dependency in pom.xml
2. Maven checks your local repository (~/.m2/repository)
3. If not found → Maven downloads from Central Repository
4. Maven stores it in your local repository
5. Maven uses it in your project
```

### Example — When Maven downloads Gson:
```
First time:
  Local repository: Not found
  → Downloads from Central Repository
  → Stores at ~/.m2/repository/com/google/code/gson/gson/2.10.1/gson-2.10.1.jar

Next time (any project):
  Local repository: Found!
  → Uses cached version (no download needed)
```

## Dependencies

A dependency is an **external library** that your project needs to work. When you add a dependency to `pom.xml`, Maven downloads it and adds it to your project's classpath.

### Adding a dependency:

```xml
<dependency>
    <groupId>com.google.code.gson</groupId>
    <artifactId>gson</artifactId>
    <version>2.10.1</version>
</dependency>
```

### Dependency Scope

Scope controls when and where the dependency is available:

| Scope | Description |
|---|---|
| `compile` | Available everywhere (default). Included in compilation, testing, and runtime. |
| `provided` | Provided by the runtime environment (like servlet-api in Tomcat). Not packaged. |
| `runtime` | Available at runtime only. Not needed for compilation. |
| `test` | Available only during testing. Not included in the final package. |
| `system` | Similar to provided, but you specify the JAR file path manually. |

### Example with scopes:

```xml
<dependencies>
    <!-- Available everywhere -->
    <dependency>
        <groupId>com.google.code.gson</groupId>
        <artifactId>gson</artifactId>
        <version>2.10.1</version>
        <scope>compile</scope>   <!-- default, can omit -->
    </dependency>

    <!-- Only for testing -->
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>5.10.0</version>
        <scope>test</scope>
    </dependency>

    <!-- Provided by the server -->
    <dependency>
        <groupId>javax.servlet</groupId>
        <artifactId>javax.servlet-api</artifactId>
        <version>4.0.1</version>
        <scope>provided</scope>
    </dependency>
</dependencies>
```

## Transitive Dependencies

When Library A depends on Library B, and Library B depends on Library C, Maven automatically downloads all three. This is called **transitive dependency management**.

```
Your Project
└── Library A
    └── Library B
        └── Library C

Maven downloads: A, B, and C automatically
```

### Example:
```xml
<!-- You add Gson -->
<dependency>
    <groupId>com.google.code.gson</groupId>
    <artifactId>gson</artifactId>
    <version>2.10.1</version>
</dependency>

<!-- Maven automatically also downloads Gson's dependencies:
     - com.google.code.findbugs:jsr305
     - org.checkerframework:checker-qual
     - etc.
-->
```

## Maven Build Lifecycle

Maven follows a **fixed sequence of phases** called the build lifecycle. When you run a Maven command, it executes all phases up to and including the phase you specified.

### Default Lifecycle Phases (in order):

```
1. validate        → Check if project structure is correct
2. compile         → Compile source code (src/main/java)
3. test            → Run unit tests (src/test/java)
4. package         → Package into JAR/WAR
5. verify          → Run integration tests
6. install         → Install JAR into local repository
7. deploy          → Deploy to remote repository
```

### What happens with each command:

```
mvn package
→ Runs: validate → compile → test → package

mvn install
→ Runs: validate → compile → test → package → verify → install

mvn deploy
→ Runs: validate → compile → test → package → verify → install → deploy
```

## Maven Plugins

Plugins are **tools that perform specific tasks** during the build lifecycle. Each phase is executed by one or more plugins.

### Common Plugins:

| Plugin | Purpose |
|---|---|
| `maven-compiler-plugin` | Compiles Java source code |
| `maven-jar-plugin` | Creates JAR files |
| `maven-war-plugin` | Creates WAR files |
| `maven-surefire-plugin` | Runs unit tests |
| `maven-install-plugin` | Installs JAR to local repository |
| `maven-deploy-plugin` | Deploys to remote repository |
| `maven-clean-plugin` | Cleans the target directory |
| `maven-resources-plugin` | Copies resources to output directory |

### Plugin configuration in pom.xml:

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.11.0</version>
            <configuration>
                <source>17</source>
                <target>17</target>
            </configuration>
        </plugin>
    </plugins>
</build>
```

## Maven Commands

### Build Commands:

| Command | What it does |
|---|---|
| `mvn clean` | Deletes the `target/` directory (removes all compiled files) |
| `mvn compile` | Compiles source code into `target/classes` |
| `mvn test` | Runs unit tests |
| `mvn package` | Creates JAR/WAR in `target/` |
| `mvn install` | Installs JAR to local repository (`~/.m2/repository`) |
| `mvn deploy` | Uploads to remote repository |

### Useful Flags:

| Flag | Description |
|---|---|
| `-DskipTests` | Skip running tests |
| `-Dmaven.test.skip=true` | Skip tests and don't compile test code |
| `-q` | Quiet mode (less output) |
| `-X` | Debug mode (verbose output) |
| `-U` | Force update of snapshots |

### Common Command Examples:

```bash
# Clean and build
mvn clean package

# Build without running tests
mvn clean package -DskipTests

# Install to local repository
mvn clean install

# Run a specific goal
mvn compiler:compile

# See full project information
mvn help:describe
```


### Step 2: Go to project directory
```bash
cd my-project
```

### Step 3: Edit pom.xml and add dependencies

### Step 4: Write your code in `src/main/java`

### Step 5: Build the project
```bash
mvn clean package
```

```bash
java -jar target/my-project-1.0.jar
```

## Maven vs Manual Dependency Management

| Feature | Manual | Maven |
|---|---|---|
| Download libraries | Manually search and download | Automatic |
| Version management | Track versions yourself | Managed in pom.xml |
| Transitive dependencies | Download each one yourself | Automatic |
| Project structure | Varies | Standardized |
| Build process | Write scripts or use IDE | `mvn package` |
| Sharing with team | Send JAR files | Share pom.xml |

---

## Key Points to Remember

- Maven automates **dependency management** and **build processes**
- `pom.xml` is the configuration file — it defines everything about your project
- Dependencies are identified by **GAV coordinates** (groupId, artifactId, version)
- Maven downloads from **Central Repository** and caches in **local repository** (`~/.m2/repository`)
- **Build lifecycle** runs phases in order: validate → compile → test → package → install → deploy
- Plugins execute each phase of the lifecycle
- Use `mvn clean package` to build, `mvn clean install` to install locally
- Always use **standard Maven project structure**
